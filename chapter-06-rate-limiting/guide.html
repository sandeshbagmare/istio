🚦 Rate Limiting — throttle the flood

Put a hard cap on how many requests a service will accept. Past the limit, callers get an instant 429 Too Many Requests — protecting you from scrapers, retry storms and abuse.

💡
The motivation. Circuit breaking (Chapter 05) reacts to concurrency. Rate limiting caps throughput — requests per unit time. It’s how you stop a single buggy client, an aggressive crawler, or a login brute-forcer from consuming all your capacity, and how you enforce fair-use tiers (“100 req/min on the free plan”).

1 What the demo did

It fired 20 rapid requests at productpage, once before and once after applying a local rate limit of “5 requests per 60 seconds”:

PhaseLimitResult (typical)
Beforenone200 × 20
Afterburst 5 / 60s200 × ~5–7 · 429 × ~13–15
🧵
The token bucket is maintained per Envoy worker thread, so the exact number allowed can be a few more than the bucket size — the throttling itself is the point. To make it exact, pin the proxy to one worker (concurrency: 1).

2 The mechanism: an EnvoyFilter token bucket

Istio has no first-class object for the local case, so you configure Envoy’s local_ratelimit filter directly via an EnvoyFilter:

token_bucket:
  max_tokens: 5          # bucket holds 5 tokens (the burst)
  tokens_per_fill: 5     # refill 5 tokens...
  fill_interval: 60s     # ...every 60 seconds
# each request spends 1 token; empty bucket -> HTTP 429

It’s attached to the productpage sidecar’s inbound listener, so every request arriving at productpage is metered.

Local rate limiting

Each proxy counts on its own — fast, no dependencies. Great for per-instance protection. (This demo.)

Global rate limiting

All proxies consult a shared rate-limit service (backed by Redis) so the quota is enforced cluster-wide. Needed for accurate “100 req/min per API key” across many replicas.

3 Verify it yourself

  1. Watch a live burst from inside the mesh: 
    kubectl exec deploy/curl-client -- sh -c "i=0; while [ $i -lt 12 ]; do curl -s -o /dev/null -w '%{http_code}\n' http://productpage:9080/productpage; i=$((i+1)); done"
    You’ll see a run of 200s, then 429s.
  2. Wait 60s for the bucket to refill, and the next requests succeed again.
  3. See the 429 in the browser: port-forward the ingress and hammer refresh: 
    kubectl port-forward -n istio-system svc/istio-ingressgateway 8080:80
    http://localhost:8080/productpage → after a few fast refreshes you get a 429 page.

4 Why this matters

  • Abuse & DoS protection: one client can’t starve everyone else.
  • Cost control: cap load on expensive downstreams (databases, third-party APIs).
  • Fair-use tiers & monetization: different quotas per plan / API key (with global RL).
  • Defense in depth: pairs with circuit breaking (concurrency) and retries/timeouts for a complete resilience story.
