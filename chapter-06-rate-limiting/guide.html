Fundamentals
01 What is Rate Limiting?

Rate limiting controls the number of requests a service accepts within a given time window. It's a critical defense mechanism that prevents any single client — or a burst of traffic — from overwhelming your services.

🔒

Protect APIs from Abuse

Prevent malicious or runaway clients from consuming all available resources by capping the request rate.

⚖️

Ensure Fair Usage

Guarantee that every client gets a fair share of capacity, preventing a noisy neighbor from starving others.

🔋

Prevent Resource Exhaustion

Safeguard CPU, memory, and downstream dependencies by rejecting excess traffic before it causes cascading failures.

📊

Operational Visibility

Track rate-limited requests through response codes (429) and custom headers to observe traffic patterns.

Comparison
02 Local vs Global Rate Limiting

Istio supports two rate limiting approaches, each suited to different use cases. In this chapter, we focus on local rate limiting using EnvoyFilter.

Aspect Local Rate Limiting Global Rate Limiting
Scope Per-pod / per-sidecar Across all pods (cluster-wide)
Mechanism EnvoyFilter with token bucket External rate limit service (e.g., Envoy ratelimit)
Latency Impact Minimal — decision is local Adds network hop to rate limit service
Accuracy Approximate (each pod counts independently) Precise (centralized counter)
Setup Complexity Low — single YAML High — requires deploying rate limit service + Redis
Best For Simple per-instance protection API quotas, multi-tenant SaaS
💡 Note: With 3 replicas and a local limit of 10 req/min per pod, the effective cluster-wide limit is approximately 30 req/min — not exactly 10.
Algorithm
03 Token Bucket Algorithm

The token bucket is the heart of Envoy's local rate limiter. Imagine a bucket filled with tokens — each incoming request consumes one token. When the bucket is empty, requests are rejected with HTTP 429. Tokens are replenished at a fixed interval.

Token Bucket — Watch tokens drain and refill
10 Max Tokens
10 Refill Amount
60s Fill Interval

How it works:
1. Bucket starts full with max_tokens (10).
2. Each request removes 1 token.
3. When tokens reach 0, subsequent requests get 429 Too Many Requests.
4. Every fill_interval (60s), the bucket is refilled with tokens_per_fill (10), up to max_tokens.

Configuration
04 EnvoyFilter Deep Dive

The EnvoyFilter resource lets you directly patch Envoy's configuration. For rate limiting, we insert the local_ratelimit HTTP filter before the router filter in the inbound filter chain.

Configuration Breakdown

1

workloadSelector

Targets only pods with app: productpage. The filter is applied exclusively to the productpage sidecar.

2

applyTo: HTTP_FILTER

Specifies we're patching at the HTTP filter level within Envoy's listener configuration.

3

match → SIDECAR_INBOUND

Applies to inbound traffic only — requests arriving into productpage, not outbound requests it makes.

4

INSERT_BEFORE router

The rate limit filter is placed before envoy.filters.http.router, ensuring rate limiting is evaluated before the request is routed to the application.

5

token_bucket Config

Defines the rate: 10 max tokens, refill 10 tokens every 60 seconds. This effectively allows 10 requests per minute per pod.

6

response_headers_to_add

Adds x-local-rate-limit: true to every response, making it easy to confirm the filter is active.

Hands-On
06 Step-by-Step Walkthrough

Follow these steps to see rate limiting in action. These match exactly what startup.bat executes.

1

Verify productpage is running

Ensure the BookInfo demo app is deployed and productpage pods are healthy.

Shell kubectl get pod -l app=productpage
2

Apply the EnvoyFilter

Deploy the local rate limit configuration targeting productpage.

Shell kubectl apply -f local-ratelimit.yaml
3

Wait for propagation

Give Envoy ~10 seconds to pick up the new filter configuration from Istiod.

Shell timeout /t 10
4

Send rapid requests

Fire 15 requests in quick succession. The first ~10 should return 200 OK, and the rest should return 429 Too Many Requests.

Shell for /L %%i in (1,1,15) do ( kubectl exec deploy/ratings-v1 -c ratings -- \ curl -s -o /dev/null -w "%%{http_code}" \ http://productpage:9080/productpage )
5

Inspect response headers

Verify the x-local-rate-limit: true header appears in the response.

Shell kubectl exec deploy/ratings-v1 -c ratings -- \ curl -s -I http://productpage:9080/productpage 2>&1 | \ findstr /i "rate-limit HTTP"

07 Full YAML Configuration

Here is the complete EnvoyFilter configuration used in this chapter, with inline annotations.

YAML apiVersion: networking.istio.io/v1alpha3 kind: EnvoyFilter metadata: name: productpage-ratelimit namespace: istio-system spec: workloadSelector: # Target specific pods labels: app: productpage configPatches: - applyTo: HTTP_FILTER # Patch HTTP filter chain match: context: SIDECAR_INBOUND # Inbound traffic only listener: filterChain: filter: name: envoy.filters.network.http_connection_manager subFilter: name: envoy.filters.http.router patch: operation: INSERT_BEFORE # Insert before router value: name: envoy.filters.http.local_ratelimit typed_config: "@type": type.googleapis.com/udpa.type.v1.TypedStruct type_url: type.googleapis.com/envoy.extensions.filters.http.local_ratelimit.v3.LocalRateLimit value: stat_prefix: http_local_rate_limiter token_bucket: # ← Core rate limit config max_tokens: 10 # Bucket capacity tokens_per_fill: 10 # Tokens added per refill fill_interval: 60s # Refill every 60 seconds filter_enabled: # Enable for 100% of traffic runtime_key: local_rate_limit_enabled default_value: numerator: 100 denominator: HUNDRED filter_enforced: # Enforce for 100% of traffic runtime_key: local_rate_limit_enforced default_value: numerator: 100 denominator: HUNDRED response_headers_to_add: # Custom response header - append_action: OVERWRITE_IF_EXISTS_OR_ADD header: key: x-local-rate-limit value: "true"
Patterns
08 API Gateway Protection

Rate limiting is one layer in a comprehensive API management strategy. Here's how it fits alongside other protection mechanisms.

🚦

Rate Limiting

Cap request volume per client/service. Prevents resource exhaustion and ensures fair usage. (This chapter)

🔌

Circuit Breaking

Halt requests to failing services to prevent cascading failures. Complements rate limiting for downstream protection.

🔐

Authentication & Authorization

Verify caller identity (JWT, mTLS) and enforce access policies (RBAC). Ensure only legitimate traffic reaches your API.

🔄

Retries & Timeouts

Configure automatic retries for transient failures and set request timeouts to avoid hanging connections.

🏗️ Defense in Depth: Combine rate limiting (Chapter 6) with circuit breaking and retry policies for robust, production-grade traffic management.
Summary
09 Key Takeaways
✓

Local rate limiting is the simplest way to protect individual service pods from traffic overload — a single EnvoyFilter YAML is all you need.

✓

The token bucket algorithm provides smooth, burst-friendly rate limiting by allowing a configurable number of tokens to be consumed per time window.

✓

EnvoyFilter patches Envoy's filter chain directly, giving you low-level control over traffic processing — but requires careful configuration.

✓

Rate-limited requests receive HTTP 429 (Too Many Requests), and custom headers like x-local-rate-limit make it easy to identify filtered responses.

✓

For precise, cluster-wide quotas, consider global rate limiting with an external service — but for per-pod protection, local rate limiting is lightweight and effective.

✓

Rate limiting works best as part of a defense-in-depth strategy combined with circuit breaking, authentication, and retry policies.
