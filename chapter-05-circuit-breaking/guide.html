What is Circuit Breaking?

Circuit breaking is a design pattern borrowed from electrical engineering. Just as an electrical circuit breaker trips to prevent a short circuit from causing a fire, a software circuit breaker trips to prevent a failing service from causing cascading failures across your entire system.

In microservices, when a downstream service becomes slow or unresponsive, upstream services can pile up requests — consuming threads, memory, and connections. Without circuit breaking, this leads to a domino effect where one failure brings down multiple services.

The Three States

🟢
Closed
Normal operation. Requests flow through freely. The breaker monitors for failures.
🔴
Open
Failure threshold exceeded. All requests are immediately rejected with 503 errors.
🟡
Half-Open
After a cooldown, limited requests are allowed through to test if the service recovered.
02

The Cascade Failure Problem

Imagine a typical microservices architecture where Service A calls Service B, which calls Service C. If Service C starts responding slowly:

  1. Service B threads pile up waiting for Service C responses
  2. Service B exhausts its connection pool and becomes unresponsive
  3. Service A threads pile up waiting for Service B
  4. Service A also becomes unresponsive
  5. The entire system is now down due to one slow service

💡 Circuit breaking stops this cascade by failing fast. Instead of waiting for a timeout, the circuit breaker immediately returns an error (503) when it detects the downstream service is struggling. This preserves resources and gives the failing service time to recover.

03

Envoy's Circuit Breaker Implementation

Istio leverages Envoy proxy's built-in circuit breaking capabilities through DestinationRule resources. The circuit breaker is configured in the connectionPool settings.

Connection Pool Settings

Parameter Description Our Value
maxConnections Maximum number of TCP connections to the destination host. Once exceeded, new connections are queued or rejected. 1
http1MaxPendingRequests Maximum number of HTTP requests that can be pending (queued) while waiting for an available connection. 1
maxRequestsPerConnection Maximum number of requests per connection. Setting to 1 disables keep-alive, ensuring a fresh connection for each request. 1

🔧 Why such low values? We intentionally set these to 1 for demonstration purposes. This makes it easy to trigger the circuit breaker with just a few concurrent requests. In production, you'd set these based on your service's actual capacity and expected load.

04

Outlier Detection

While connection pool settings control how many requests can flow, outlier detection monitors the health of upstream hosts and ejects unhealthy ones from the load balancing pool.

Parameter Description Our Value
consecutive5xxErrors Number of consecutive 5xx errors before a host is ejected from the pool. 1
interval How often the ejection analysis runs. The proxy checks host health at this interval. 1s
baseEjectionTime Minimum duration a host remains ejected. Actual time increases with consecutive ejections. 3m
maxEjectionPercent Maximum percentage of hosts that can be ejected. 100% means all hosts can be ejected if unhealthy. 100

Step-by-Step Walkthrough

Follow along with startup.bat to see circuit breaking in action.

1
Deploy httpbin Service
Deploy the target httpbin service that we'll be sending requests to. This is a simple HTTP echo service that returns information about requests.
2
Deploy Fortio Load Tester
Deploy Fortio, a powerful load testing tool from Istio's ecosystem. It can generate controlled amounts of concurrent traffic and report detailed statistics.
3
Apply Circuit Breaker DestinationRule
Apply the DestinationRule that configures the circuit breaker with maxConnections: 1, http1MaxPendingRequests: 1, and outlier detection settings.
4
Demo 1: Single Request (Baseline)
Send a single request to verify basic connectivity. With just one connection, we stay within the circuit breaker limits, and the request succeeds with HTTP 200.
kubectl exec $FORTIO_POD -c fortio -- \ fortio curl http://httpbin:8000/get
5
Demo 2: 2 Concurrent Connections
Send 20 requests with 2 concurrent connections. Since maxConnections=1, the second connection will overflow and some requests will be rejected with HTTP 503.
kubectl exec $FORTIO_POD -c fortio -- \ fortio load -c 2 -qps 0 -n 20 \ -loglevel Warning http://httpbin:8000/get
6
Demo 3: 3 Concurrent Connections
Send 20 requests with 3 concurrent connections. With even more connections exceeding the limit, you'll see a higher ratio of 503 errors — demonstrating proportional circuit breaking.
kubectl exec $FORTIO_POD -c fortio -- \ fortio load -c 3 -qps 0 -n 20 \ -loglevel Warning http://httpbin:8000/get
7
Verify with Envoy Stats
Query the Envoy sidecar's statistics to see the upstream_rq_pending_overflow counter. This shows exactly how many requests were rejected by the circuit breaker.
kubectl exec $FORTIO_POD -c istio-proxy -- \ pilot-agent request GET stats | grep upstream_rq_pending_overflow

YAML Configurations

Circuit Breaker DestinationRule

The core configuration — this DestinationRule applies circuit breaking to the httpbin service.

apiVersion: networking.istio.io/v1 kind: DestinationRule metadata: name: httpbin-circuit-breaker spec: host: httpbin trafficPolicy: connectionPool: tcp: maxConnections: 1 http: http1MaxPendingRequests: 1 maxRequestsPerConnection: 1 outlierDetection: consecutive5xxErrors: 1 interval: 1s baseEjectionTime: 3m maxEjectionPercent: 100

Key Takeaways

⚡ Fail fast, recover faster. Circuit breaking immediately rejects excess requests instead of letting them queue up and consume resources. This gives the overloaded service breathing room to recover.
🔧 No code changes required. Istio's circuit breaking is configured entirely through DestinationRule resources. Your application code doesn't need any circuit breaker libraries — Envoy handles it transparently.
📊 Proportional response. More concurrent connections beyond the limit leads to more requests being circuit-broken. The protection scales with the level of overload.
🔍 Observable and measurable. Envoy exposes detailed statistics like upstream_rq_pending_overflow so you can monitor exactly how often circuit breaking activates and tune your thresholds.
🛡️ Two layers of protection. Connection pool limits prevent overloading with too many connections, while outlier detection ejects unhealthy hosts that return errors. Together, they provide comprehensive resilience.
⚙️ Tune for production. Our demo uses aggressive values (maxConnections: 1) for easy demonstration. In production, set values based on your service's actual capacity, expected load, and SLO requirements.
