Understanding

What is Kiali?

Kiali is the observability console for Istio service mesh. It provides deep insight into the structure and health of your mesh, helping you understand what your microservices are doing in real time.

Why Kiali Matters

Without Kiali, understanding the behavior of dozens or hundreds of microservices communicating through Istio's sidecar proxies would require manual inspection of metrics, logs, and configuration files across many systems. Kiali unifies all of this into a single, interactive dashboard.

🔍

Topology Visualization

Automatically generates real-time service mesh topology graphs showing how services communicate with each other.

❤️

Health Monitoring

Color-coded health indicators for namespaces, applications, workloads, and services at a glance.

✅

Config Validation

Validates Istio configurations (VirtualServices, DestinationRules) and warns about misconfigurations before they cause issues.

📊

Metrics Integration

Pulls metrics from Prometheus to show request rates, error rates, and latency distributions for every service.

🔒

mTLS Verification

Visually verifies that mutual TLS is properly configured between services with lock icons on traffic edges.

🔗

Distributed Tracing

Integrates with Jaeger to provide distributed tracing capabilities directly within the Kiali interface.

Setup

Getting Started

Follow these steps to access Kiali and start visualizing your service mesh traffic.

1

Verify Kiali is Installed

Check that Kiali is running in the istio-system namespace.

bash 
kubectl get pods -n istio-system -l app=kiali

You should see a kiali pod in Running state.

2

Set Up Port Forwarding

Forward the Kiali service port to your local machine to access the dashboard.

bash 
kubectl port-forward svc/kiali -n istio-system 20001:20001
3

Generate Traffic

Kiali needs active traffic to display graphs. Apply the traffic generator job to create steady request flow.

bash 
kubectl apply -f generate-traffic.yaml

# This sends 5 requests/second for 300 seconds
# with 2 concurrent connections to productpage
4

Open Kiali Dashboard

Navigate to the Kiali web interface in your browser.

url 
http://localhost:20001

Select the Graph view from the sidebar, choose the default namespace, and select Versioned App Graph to see traffic flowing through your services.

💡

Tip: The startup.bat script automates all these steps for you! Just double-click it to deploy traffic, start port-forwarding, and open this guide.

Exploration

Kiali UI Deep Dive

Kiali's interface is organized into several views, each focused on a different aspect of your service mesh. Let's explore each one in detail.

📋 Overview Page

The Overview page is your mesh-wide dashboard. It shows a card for each namespace that is part of the mesh, giving you an at-a-glance summary of health and traffic.

What You'll See

  • ▸ Namespace Health: Aggregated health status (green/yellow/red) for all apps, workloads, and services in each namespace
  • ▸ Traffic Summary: Sparkline graphs showing inbound/outbound request rates for the namespace
  • ▸ Istio Config Status: Number of valid, warning, and error Istio configurations
  • ▸ Quick Navigation: Click any namespace card to jump directly to its graph, apps, or workloads view

🕸️ Graph View — The Heart of Kiali

The Graph view is Kiali's most powerful feature. It renders a real-time, interactive topology graph of your service mesh, showing how services communicate and the health of each connection.

Graph Elements

⬡

Nodes

Represent apps, workloads, or services. Shape indicates type: app (triangle), workload (circle), service (pentagon). Color indicates health.

→

Edges

Represent traffic between nodes. Thickness indicates request volume. Color indicates health: green (success), red (errors).

🎬

Traffic Animation

Animated dots flow along edges showing real-time traffic direction and volume. Faster animation = higher request rate.

📊

Response Codes

Hover over edges to see HTTP response code distribution (2xx, 3xx, 4xx, 5xx) and percentage breakdown.

Versioned App Graph (Recommended)

The Versioned App Graph groups nodes by application label and shows different versions (e.g., reviews-v1, reviews-v2, reviews-v3) as separate nodes within an application box. This is the ideal view for understanding traffic routing between versions — essential when testing canary deployments or A/B testing.

Graph Interpretation 
productpage ──→ reviews (app box)
                     ├── reviews-v1  (33% traffic)
                     ├── reviews-v2  (33% traffic) ──→ ratings-v1
                     └── reviews-v3  (33% traffic) ──→ ratings-v1

productpage ──→ details-v1

⚙️ Display Options

Kiali's graph view offers rich display customization through the toolbar. These options control what information is overlaid on the topology graph.

🎬

Traffic Animation

Toggle animated dots flowing along edges to visualize real-time request flow. Speed corresponds to request volume.

⏱️

Response Time

Display average response time labels on edges. Helps identify slow service-to-service calls at a glance.

📶

Throughput

Show requests-per-second on each edge. Useful for understanding load distribution across services.

🔒

mTLS Locks

Display lock icons on edges where mutual TLS is active. Open locks indicate missing mTLS — a security concern.

💡

Pro Tip: Enable Traffic Animation and Response Time together for the most informative graph view. This combination lets you see both the flow pattern and performance characteristics simultaneously.

📱 Applications View

The Applications view groups workloads by the app Kubernetes label. This gives a higher-level view than individual workloads, showing you application-level health and metrics.

Application Details

  • ▸ Health Status: Aggregated health from all workloads and services belonging to the application
  • ▸ Inbound/Outbound Metrics: Request rate, error rate, and latency (P50/P90/P99) charts for each application
  • ▸ Workload List: Shows all workloads (deployments) that belong to this application with their individual status
  • ▸ Traffic Tab: Visual graph scoped to just this application's inbound and outbound connections

⚙️ Workloads View

Dive into individual workloads (Deployments, StatefulSets, DaemonSets) for granular inspection of pods, logs, and Envoy proxy configuration.

Workload Details

  • ▸ Pod Details: View individual pod status, labels, annotations, and container information
  • ▸ Logs: Stream and search application container and istio-proxy sidecar logs directly from the UI
  • ▸ Envoy Config: Inspect the Envoy proxy configuration dumped from the sidecar — clusters, listeners, routes, and bootstrap
  • ▸ Metrics: Workload-level inbound/outbound metrics with configurable time ranges
💡

Debugging Tip: When troubleshooting, use the Envoy tab to inspect the actual proxy configuration. This is invaluable for understanding why traffic routing isn't behaving as expected — you can see exactly what routes, clusters, and listeners Envoy has configured.

🌐 Services View

The Services view focuses on Kubernetes Services and their associated Istio routing configuration.

Service Details

  • ▸ Virtual Services: View all VirtualService resources that route traffic to this service, including match conditions, destinations, and weights
  • ▸ Destination Rules: See DestinationRule configurations including traffic policies, connection pool settings, and outlier detection
  • ▸ Endpoints: View the actual pod endpoints backing the service
  • ▸ Metrics: Service-level request rate, error rate, and latency charts

🛡️ Istio Config View

The Istio Config view is a centralized place to inspect and validate all Istio custom resources in your cluster.

Configuration Status

✅

Valid

Configuration is syntactically correct and semantically valid. References to hosts, subsets, and gateways all resolve properly.

⚠️

Warning

Configuration has potential issues — e.g., a VirtualService references a subset that doesn't exist in any DestinationRule yet.

❌

Error

Configuration has definite errors that will prevent correct mesh behavior — e.g., conflicting VirtualServices or invalid syntax.

⚠️

Important: Always check the Istio Config view after applying new VirtualServices or DestinationRules. Kiali's validation catches many common mistakes like missing subsets, host mismatches, and conflicting route rules that kubectl apply won't flag.

Visualization

Graph Types Explained

Kiali offers four graph types, each providing a different perspective on your service mesh traffic.

Graph Type Grouping Best For Status
App Graph Groups by app label High-level service overview Standard
Versioned App Graph Groups by app + version Canary deployments, traffic splitting Recommended
Workload Graph Individual deployments Detailed pod-level traffic analysis Detailed
Service Graph Kubernetes Services Service-oriented architecture view Simplified
💡

Recommendation: Start with the Versioned App Graph. It provides the best balance between detail and clarity, especially when working with the BookInfo sample application where multiple versions of the reviews service exist. You can see exactly how traffic splits between v1, v2, and v3.

Monitoring

Health Indicators

Kiali uses a consistent color-coding system across all views to communicate health status at a glance.

Healthy (Green)

All requests succeeding. Error rate is below the configured threshold (typically <0.1%).

Degraded (Yellow)

Some errors detected. Error rate is above warning threshold but below failure threshold.

Failure (Red)

High error rate. Significant portion of requests are failing (5xx errors). Immediate attention needed.

No Health (Gray)

No traffic detected in the selected time period. Service may be idle or not receiving requests.

Where Health Appears

Health indicators are consistently shown across all Kiali views:

  • ▸ Overview: Namespace-level aggregated health
  • ▸ Graph: Node colors and edge colors reflect health of individual components and connections
  • ▸ List Views: Small health badge icons next to each application, workload, and service
Troubleshooting

Identifying Issues with Kiali

Kiali makes it easy to spot problems in your mesh. Here are the key patterns to watch for.

  • Red Edges → 5xx Server Errors Red-colored edges in the graph indicate HTTP 5xx errors on that connection. Click the edge to see the exact error rate percentage and response code breakdown. Common causes: crashing pods, resource exhaustion, or misconfigured retries.
  • Slow Response Times Enable "Response Time" in display options. Look for edges with high latency values (e.g., >500ms). This often indicates slow database queries, resource contention, or misconfigured timeouts in VirtualServices.
  • Missing mTLS (Open Lock Icons) When mTLS display is enabled, edges without mTLS show an open lock icon. This means traffic between those services is not encrypted. Check your PeerAuthentication and DestinationRule TLS settings.
  • Misconfigured VirtualServices Check the Istio Config view for warning/error badges. Common issues include VirtualServices referencing non-existent subsets, weight distributions not summing to 100, or overlapping route match conditions.
💡

Debugging Workflow: Start from the Graph View to identify the problem area → click the affected edge or node → check Metrics for the error timeline → inspect Workload Logs for root cause → validate Istio Config for misconfigurations.

Power User

Useful Features

Beyond basic visualization, Kiali offers several powerful features for managing and debugging your mesh.

📜

Traffic Policies

View and understand how VirtualServices and DestinationRules affect traffic flow. See active routing rules overlaid on the graph.

🔗

Tracing Integration

Click any service and jump directly to its distributed traces in Jaeger. Correlate mesh topology with request-level traces.

🛡️

Config Validation

Kiali validates all Istio resources and highlights conflicts, missing references, and semantic errors that kubectl won't catch.

⏪

Replay

Use the time range selector to view historical traffic patterns. Compare how traffic flow and health changed over time.

🔍

Filtering & Search

Filter graph nodes by app, version, or workload name. Hide idle nodes or show only nodes with errors for focused debugging.

📐

Layout Options

Choose between different graph layout algorithms (dagre, cola, klay) to best visualize your specific topology structure.

🎯 Key Takeaways

  • Kiali is the primary observability console for Istio, providing real-time visualization of your entire service mesh topology and health.
  • The Versioned App Graph is the most useful graph type for understanding traffic routing between service versions — essential for canary and A/B testing scenarios.
  • Health indicators use a consistent green/yellow/red/gray color scheme across all views — green means healthy, red means immediate attention needed.
  • Kiali validates Istio configurations and catches mistakes that kubectl apply won't flag, like missing subsets, conflicting routes, and host mismatches.
  • The debugging workflow should start with the Graph View to identify problem areas, then drill down into metrics, logs, and config validation.
  • Enable Traffic Animation and mTLS lock icons in the display options for the most informative graph visualization.
  • Kiali integrates with Prometheus (metrics), Grafana (dashboards), and Jaeger (tracing) — making it a unified hub for all observability needs.
  • Always generate traffic before opening Kiali — the graph requires active traffic to display meaningful data.
