💥 Fault Injection — break things on purpose

Force errors and delays into a healthy service from config alone, so you can prove your app survives a bad dependency before a real outage does the testing for you.

💡
The motivation. Every distributed system will have a slow or failing dependency eventually. Do your timeouts fire? Do retries help or make it worse? Does one slow service stall the whole page? You can’t answer that from a diagram — you have to test failure. Traditionally you’d hack failure code into the app. Istio lets you inject faults externally, precisely, and remove them just as easily.

1 What the demo did

Against a perfectly healthy ratings service, the script injected two classic faults and measured the effect from inside the mesh:

PhaseInjected faultObserved
Baselinenone200, a few ms
Demo 1 · Abort50% → HTTP 500~half the calls 500
Demo 2 · Delay100% → +3severy call ~3.0 s

The ratings code never changed — Envoy generated the errors and the latency.

2 The two fault types

Abort — inject errors

http:
- fault:
    abort:
      percentage: {value: 50}
      httpStatus: 500
  route:
  - {destination: {host: ratings, subset: v1}}

Great for testing error handling, fallbacks and alerting.

Delay — inject latency

http:
- fault:
    delay:
      percentage: {value: 100}
      fixedDelay: 3s
  route:
  - {destination: {host: ratings, subset: v1}}

Great for testing timeouts, retries and slow-dependency behavior.

🎯
You can scope faults precisely — e.g. only for end-user: jason (combine with a match block from Chapter 03) — so you chaos-test in production affecting only yourself.

3 See the user-facing impact

  1. Apply the delay fault (if not already active): 
    kubectl apply -f fault-delay.yaml
  2. Port-forward & open the app: 
    kubectl port-forward -n istio-system svc/istio-ingressgateway 8080:80
    http://localhost:8080/productpage
  3. The Book Reviews panel shows “Ratings service is currently unavailable” — because reviews v2/v3 have a shorter timeout than your injected 3s delay. That’s the exact failure mode you want to discover in a drill, not an incident.

4 Why this matters

  • Confidence, not hope: verify resilience mechanisms actually work under failure.
  • Game days / chaos drills: reproducible, config-driven failures you can run in staging or prod.
  • Safe blast radius: target a % or a specific user, and remove the fault in one command.
  • Pairs with Chapter 05: inject failures, then watch a circuit breaker eject the bad host automatically.
