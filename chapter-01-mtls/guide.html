Security · Zero-Trust Networking

🔐 Mutual TLS — encrypt everything, change nothing

Turn on automatic, certificate-based encryption and identity for every call between your microservices — without touching a single line of application code.

💡
The motivation — why mTLS exists.
Inside a Kubernetes cluster, pods talk to each other over plain HTTP by default. Anyone who lands on the network (a compromised pod, a curious sidecar, a misconfigured tool) can sniff that traffic or impersonate a service. The old fix was to make every team add TLS certificates to their app — slow, error-prone, and rarely done. Istio moves that job out of the app and into the mesh: the sidecar proxy handles the crypto for you.

1 What actually happened when you ran the demo

The startup script deployed two identical curl clients and had each of them call the ratings service twice — once before and once after STRICT mTLS was switched on:

ClientHas Istio sidecar?Before STRICTAfter STRICT
curl-mesh (namespace default)✅ Yes HTTP 200HTTP 200
curl-legacy (namespace legacy)❌ No HTTP 200HTTP 000 — blocked

The only thing that changed was a policy object. The apps were never redeployed.

2 The one object that did it

A PeerAuthentication in STRICT mode tells every Envoy sidecar in the namespace: “only accept connections that present a valid Istio identity.”

apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: default-strict-mtls
  namespace: default
spec:
  mtls:
    mode: STRICT        # PERMISSIVE (default) | STRICT | DISABLE
PERMISSIVE accept both plaintext & mTLS (safe migration default)  ·  STRICT mTLS only  ·  DISABLE turn mTLS off

3 How it works under the hood

Identity, not passwords

When a pod starts, istiod issues its sidecar a short-lived X.509 certificate whose “name” is a SPIFFE ID derived from the Kubernetes service account, e.g. spiffe://cluster.local/ns/default/sa/bookinfo-ratings. That certificate is the workload’s cryptographic identity.

Automatic & rotating

The sidecars perform a mutual TLS handshake on both ends (client and server prove who they are). Certificates auto-rotate about every 24h. Your application still thinks it is speaking plain HTTP to localhost — the sidecar does the encryption.

your-app ──plain http──▶ [Envoy sidecar] ══ mTLS (encrypted) ══▶ [Envoy sidecar] ──plain http──▶ ratings-app

4 See it & verify it yourself

Open a terminal (PowerShell or Git Bash) in this folder and try these:

  1. Confirm the policy is active 
    kubectl get peerauthentication -n default
  2. Re-run the blocked call — the legacy client still fails 
    kubectl exec -n legacy deploy/curl-legacy -- curl -s -o /dev/null -w "%{http_code}\n" --max-time 6 http://ratings.default:9080/ratings/0
    Returns 000 (connection reset by the server’s sidecar).
  3. Prove traffic really is mTLS — inspect the certificate Istio issued to a pod 
    ..\tools\bin\istioctl.exe proxy-config secret deploy/reviews-v1 -o json
    (You’ll see the X.509 chain the sidecar presents. On macOS/Linux drop the .exe.)
✅
There is no dashboard to open for this chapter — the proof is in the HTTP status codes the script printed and the commands above. Chapters 07–09 add the visual dashboards (Kiali even draws a 🔒 lock icon on every mTLS-secured edge).

5 Why this matters

  • Zero-trust by default: traffic is encrypted and authenticated even inside the cluster.
  • No app changes: security is a platform capability, not a per-team chore.
  • Foundation for authorization: once every call carries a verifiable identity, you can write rules like “only productpage may call reviews” (AuthorizationPolicy).
  • Compliance: “encryption in transit” is satisfied mesh-wide with one object.
