🐤 Canary Deployments — ship new versions safely

Send a tiny slice of live traffic to a brand-new version, watch it, and grow the slice only when you trust it — all by editing a single weight number.

💡
The motivation. A “big bang” deploy — flip 100% of users to a new version at once — is terrifying: a bug hits everyone instantly. The safer idea, borrowed from coal miners’ canaries, is to expose the new version to a small % of traffic first. Historically that meant running two deployments behind a load balancer and fiddling with replica counts. Istio makes the traffic split a declarative percentage that is completely independent of how many pods you run.

1 What the demo did

The Bookinfo reviews service has three versions deployed side by side:

v1 — no rating stars v2 — black stars v3 — red stars

The script walked a full canary rollout, sampling 50 live requests at each stage so you can see the split move:

StageVirtualService weightsWhat you should see
Step 0 — baselinev1: 100all requests → v1
Step 1 — canaryv1: 90 · v3: 10~9 of 10 → v1, ~1 of 10 → v3
Step 2 — widenv1: 50 · v3: 50roughly half and half
Step 3 — promotev3: 100all requests → v3
🎲
Weights are probabilistic per request, not a strict quota — with only 50 samples you’ll see close-to but not exactly 90/10. Over thousands of requests it converges to the configured ratio.

2 The mechanism: DestinationRule + VirtualService

DestinationRule = the menu

Defines named subsets by pod label, so you can refer to “v1” or “v3”. Applied once by setup:

kind: DestinationRule
spec:
  host: reviews
  subsets:
  - {name: v1, labels: {version: v1}}
  - {name: v3, labels: {version: v3}}

VirtualService = the split

Assigns a percentage of traffic to each subset. This is the only thing the demo changed:

kind: VirtualService
spec:
  hosts: [reviews]
  http:
  - route:
    - {destination: {host: reviews, subset: v1}, weight: 90}
    - {destination: {host: reviews, subset: v3}, weight: 10}

3 See it with your own eyes

  1. Open the app. In a new terminal: 
    kubectl port-forward -n istio-system svc/istio-ingressgateway 8080:80
    then browse to http://localhost:8080/productpage
  2. Set a 50/50 split and hammer refresh — the “Book Reviews” star color flips between none (v1) and red (v3) roughly half the time: 
    kubectl apply -f reviews-canary-50-50.yaml
  3. Re-run the numeric sampler at any weight: 
    kubectl exec deploy/curl-client -- sh -c "for i in $(seq 1 20); do curl -s http://reviews:9080/reviews/0 | grep -o reviews-v[0-9]; done | sort | uniq -c"
  4. Best view of all — Kiali (Chapter 07). Its graph draws two live edges from reviews with the request % on each. That’s the picture-perfect canary visualization.

4 Why this matters

  • Risk control: a bad release hurts 10% of users, not 100% — and you roll back by re-applying the old weights.
  • Decoupled from scaling: the 10% is a routing decision, unrelated to replica counts. 1 canary pod can still take exactly 10%.
  • Foundation for automation: tools like Argo Rollouts / Flagger automate exactly this — bump the weight, check metrics, bump again, or roll back.
  • A/B & blue-green are the same primitive with different weights or match rules (see Chapter 03).
