🎯 Content-Based Routing — send the right user to the right version

Route requests by who they come from — a header, a cookie, a device, a path — instead of a blind percentage. This is how dark launches and A/B tests work.

💡
The motivation. Canary (Chapter 02) is random: you can’t control which 10% of users become canaries. But often you want specific people on the new version — your QA team, internal staff, a beta cohort, or users in a certain region — while everyone else stays on stable. That needs routing based on request content, not dice rolls.

1 What the demo did

It applied a rule: “if the request says you are jason, serve reviews v2; otherwise serve v1.” Then it fired 10 requests each way from inside the mesh:

RequestHeaderServed by
Anonymous(none)v1 — no stars
Logged in as jasonend-user: jasonv2 — black stars

Same service, same URL — the version is decided entirely by the request’s identity.

2 The rule (evaluated top-to-bottom)

kind: VirtualService
spec:
  hosts: [reviews]
  http:
  - match:                       # RULE 1
    - headers:
        end-user:
          exact: jason           #   if header end-user == jason
    route:
    - {destination: {host: reviews, subset: v2}}
  - route:                       # RULE 2 (default / fallthrough)
    - {destination: {host: reviews, subset: v1}}
🔎
Order matters. Istio tries each match in sequence and uses the first one that fits. The last route with no match is the catch-all — always put it last.

Where does end-user come from? When you log into Bookinfo, the productpage service attaches an end-user header to its calls. Istio reads that header at the reviews hop and routes accordingly.

3 You can match on almost anything

headerscookies uri prefix / exact / regexmethod query paramssource workload/namespace 
# mobile users -> v3 ; /api/v2 path -> a new backend ; canary cookie -> v2
- match: [ {headers: {user-agent: {regex: ".*Mobile.*"}}} ]
- match: [ {uri: {prefix: "/api/v2"}} ]
- match: [ {headers: {cookie: {regex: "^(.*;?)?(canary=true)(;.*)?$"}}} ]

4 See it in the browser

  1. Port-forward the ingress in a new terminal: 
    kubectl port-forward -n istio-system svc/istio-ingressgateway 8080:80
  2. Open http://localhost:8080/productpage — the reviews panel shows no stars (that’s v1, the anonymous default).
  3. Click Sign in (top-right) and log in as jason (any password). The reviews panel now shows black stars — you’ve been routed to v2.
  4. Sign in as any other name (e.g. alice) → back to v1. Only jason is special.

5 Why this matters

  • Targeted rollouts: put internal/QA users on the new build in production, safely, before anyone else.
  • True A/B testing: deterministically bucket users and compare business metrics per version.
  • Migrations: route /api/v2/* to a new service while /api/v1/* stays put — a strangler-fig migration with zero client changes.
  • Combine with weights: a match block can also carry weighted destinations — “jason’s traffic, split 50/50 between v2 and v3.”
